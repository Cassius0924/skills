# PR 改动总结生成器

根据分支间的 git diff，生成结构化、分优先级的改动总结，便于 Reviewer 快速理解 PR 的整体改动。

## 工作流程

### 第一步：获取 Diff 数据

使用 `scripts/get_diff.py` 脚本自动获取分支间的差异。这个脚本会先拉取 `--stat` 概览，再逐文件获取详细 diff，避免一次性加载过多内容。

```bash
python {SKILL_DIR}/scripts/get_diff.py [OPTIONS]
```

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `--source` | 源分支 | 当前分支 (HEAD) |
| `--target` | 目标分支 | 自动检测 main/master |
| `--max-lines` | 每个文件最大 diff 行数 | 300 |
| `--output` | 输出格式 (text / json) | text |

**运行示例：**

```bash
# 默认：当前分支 vs 自动检测的 main/master
python {SKILL_DIR}/scripts/get_diff.py

# 指定目标分支
python {SKILL_DIR}/scripts/get_diff.py --target develop

# 大量改动时增加行数上限
python {SKILL_DIR}/scripts/get_diff.py --max-lines 500
```

如果用户未指定源分支或目标分支，不要追问，直接运行脚本让它自动检测。只有脚本报错（比如找不到 main/master）时才需要询问用户。

### 第二步：分析 Diff 并生成总结

拿到 diff 输出后，按以下策略分析和组织内容。

#### 分析顺序

1. **先看 stat 概览** — 了解改动规模、涉及哪些模块/目录
2. **再看逐文件 diff** — 理解每个改动的具体意图
3. **归纳合并** — 同一目的的跨文件改动合并为一个点（例如：重命名某函数涉及 5 个文件，只算一个改动点）
4. **按重要性排序** — 业务逻辑 > 架构变更 > bug 修复 > 性能优化 > 工具/配置/格式化

#### 输出规则

**数量控制：**
- 归纳后的改动点 ≤ 5 个：直接按优先级列出
- 归纳后的改动点 > 5 个：拆为「重点改动」（≤ 5 个）和「其他改动」（尽量精简合并）

**内容要求：**
- 每个改动点一句话说清 **改了什么 + 为什么改**，不需要重复文件的完整 diff 内容
- 关键文件名用括号附在句尾
- 纯格式化、import 排序、空行调整等无实质影响的改动，如果没有其他内容则可忽略，如果有则归入「其他改动」用一句话带过
- 如果有文件 diff 被截断，在末尾加一句说明

**语言：** 跟随用户的对话语言。用户用中文就中文输出，用户用英文就英文输出。

#### 输出模板

**改动点 ≤ 5 个时：**

```
## PR 改动总结

**分支**: {source} → {target}

1. 新增用户鉴权中间件，支持 JWT 令牌校验 (`middleware/auth.go`)
2. 重构订单查询接口，N+1 查询优化为批量加载 (`service/order.go`)
3. 修复并发场景下库存扣减的竞态条件 (`repo/inventory.go`)
```

**改动点 > 5 个时：**

```
## PR 改动总结

**分支**: {source} → {target}

### 重点改动

1. ...
2. ...

### 其他改动

- ...
- ...
```

### 注意事项

- 不要逐文件机械翻译 diff，要提炼改动意图
- 如果 diff 内容为空（两个分支没有差异），直接告知用户「两个分支之间没有差异」
- 如果脚本运行失败（如不在 git 仓库中），将错误信息告知用户并给出建议
